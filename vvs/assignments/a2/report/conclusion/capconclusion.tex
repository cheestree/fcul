\chapter{Conclusion} \label{cap:conclusion}

The application already implements the main functionality expected from a small meeting scheduler: users can register, propose meetings, respond to invitations, view their calendar, copy discovered events, and export a personal iCalendar feed. The existing end-to-end tests confirm that the central happy path works across the web, service, and persistence layers.

However, the inspection also showed that important business rules are not yet fully protected. The most significant issue is the lack of enforced conflict detection, even though repository support for overlapping meetings already exists. Input validation and iCalendar compliance also need further attention.

Future work should therefore prioritize service-level tests and fixes before adding many more interface-level tests. This would make the application more robust while keeping the test suite fast and focused. After those rules are protected, the report and tests can be expanded with more detailed coverage of discovery providers, calendar export compatibility, and negative user workflows.
