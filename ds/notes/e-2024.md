#### 1. Evaluating whether a software system has a “good” design involves measurable indicators and principles that provide objective criteria for assessment

##### a. What measurable indicators can be used to evaluate if a software design is "good"?
>
>Some indicators include scalability metrics, maintainability index, code quality as well as performance and usability metrics.

##### b. How do software design principles, such as separation of concerns or abstraction, contribute to assessing design quality?
>
> Separation of concerns allows to isolate different aspects of a system, making it easier to manage complexity, enhance maintainability, and facilitate collaboration among development teams. By separating concerns, changes in one part of the system have minimal impact on other parts, leading to a more robust and adaptable architecture.

#### 2. Performance is a critical quality attribute in software systems

##### a. Explain what it consists of
>
> Performance is the ability of a system to respond quickly to user inputs or process data efficiently. It is crucial for user satisfaction and overall system effectiveness, especially in applications where speed and responsiveness are key.

##### b. In general, requirements for different quality attributes often conflict with one another. Provide an example of a software system that illustrates potential conflicts between performance and other quality attribute
>
> Performance and security are often conflicting, as one may require additional processing time to ensure data protection, such as encryption and authentication mechanisms, which can slow down system responsiveness.

### G2

#### SmartSuggest is an e-commerce recommendation system that aims to provide personalized product recommendations to users based on their browsing and purchasing behavior. The system collects real-time user activity data (e.g., product views, clicks, and purchases). It must generate personalized recommendations for each active user in near real-time and periodically produce global trending product lists (daily, weekly, monthly). The recommendations must prioritize performance and scalability to handle thousands of users. The system must allow easy updates to recommendation algorithms and support integration with external analytics tools

#### 1. Choose an architectural style suitable for this system

##### a.  Suggest an architectural style suitable for this system, considering the need for scalability and independent deployment of services. Describe, in abstract terms, the types of components and connectors used. Explain the key advantages and disadvantages of the identified style
>
>A client-server architecture would benefit this type of system the most, as the client allows for a user-friendly interface for costumers to browse and purchase products, while the server handles product suggestions, inventory management, and payment processing. This separation of concerns allows for scalability, as the server can handle multiple clients simultaneously, and it also facilitates maintenance and updates to either the client or server independently. Scaling servers can be done based on demand, ensuring that the system remains responsive during peak shopping times, while clients can be lightweight applications or web interfaces that run on various devices, as well as the increase of database capacity to handle more products and user data. A pub-sub implementation can also be used to handle real-time updates if needed. However, potential challenges include complexity in managing server load and balancing performance across different client devices, as well as coding security measures to protect sensitive user information during transactions.

#### 2. Why is it essential to use multiple architectural views to document a system like SmartSuggest? Provide examples of views that would be relevant and explain their purpose
>
>The use of multiple views is necessary because different stakeholders have different concerns and requirements. For example, developers may focus on the implementation view, while business stakeholders may be more interested in the process view. By providing multiple views, the architecture can effectively communicate its structure and behavior to all stakeholders, ensuring that their concerns are addressed and facilitating better decision-making throughout the development process.

#### 3. Imagine the module responsible for product recommendations needs to be outsourced to an external vendor. What are the advantages of using the architecture identified in 1 (a) compared to other tightly coupled architectures?
>
>A microservice focused client-server architecture has the big advantage of scalability and modularity, allowing independent deployment and scaling of services based on demand. If one microservice experiences high load, it can be scaled independently without affecting the entire system. This architecture also enhances fault isolation, as failures in one service do not necessarily impact others.

#### 1. It is well known that a system's architecture strongly influences different quality attributes

##### a. Explain the typical structure of a quality scenario and discuss the advantages of using these scenarios to express requirements related to quality attributes
>
>A quality scenario has six parts: source (who/what triggers), stimulus (the event), environment (system state), artifact (affected component), response (what happens), response measure (quantifiable metric). They provide several advantages like requirements testable, clear communication, design guidance, validation criteria, prioritization, and evaluation basis with non-technical stakeholders.
>
>Source of Stimulus: User requests for product recommendations.
>Stimulus: A user browses products on the e-commerce platform.
>Environment: Normal operation with average load (10000 users).
>Artifact: Recommendation engine module.
>Response: The recommendation engine processes the user's browsing history and preferences to generate personalized product recommendations.
>Measured Response: Time taken to generate and display recommendations, accuracy of recommendations based on user feedback, down from 0.5s to 0.1s for example.
>
>With the system being real-time, the recommendation engine should be able to process user data and generate recommendations quickly, ideally within a few seconds, to ensure a seamless user experience.

>Modifiability Scenario:
>Source of Stimulus: Change in recommendation algorithm.
>Stimulus: A new machine learning algorithm is introduced to improve recommendation accuracy.
>Environment: Normal operation.
>Artifact: Recommendation engine module.
>Response: The recommendation engine is updated to incorporate the new algorithm without affecting other system components.
>Measured Response: Time taken to implement the change in 2 working days with 0 impact on system performance and no impact to other modules. >Expected load of 15000 users and 500 requests per second.

>

However, these strategies are a temporary fix, and the underlying architecture should be designed to handle scalability effectively.

##### b. What are the advantages of managing SmartSuggest as a Software Product Line instead of developing independent systems for each configuration?
>
>Managing SmartSuggest as a Software Product Line (SPL) offers several advantages over developing independent systems for each configuration. First, it promotes reusability, as common features and components can be shared across different configurations, reducing development time and effort. Second, it enhances maintainability, as updates and bug fixes can be applied to the core product line, benefiting all configurations simultaneously. Third, it allows for faster time-to-market, as new configurations can be created by selecting and combining existing features rather than building from scratch. Additionally, SPLs facilitate better resource allocation and cost efficiency, as development teams can focus on enhancing the core product line rather than duplicating efforts for each independent system.
