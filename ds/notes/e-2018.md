# Design de Software

## Exame (2017-2018)

### G1

#### 1

a) A maleabilidade representa a capacidade de um sistema de software se adaptar a mudanças nos requisitos ou no ambiente operacional sem a necessidade de grandes modificações na arquitetura ou no código. Isso é importante porque permite que o software evolua ao longo do tempo, atendendo às novas necessidades dos usuários e às mudanças tecnológicas, sem incorrer em altos custos de manutenção ou riscos de falhas.

b) Technical Debt é um conceito que se refere ao custo acumulado de decisões de design ou implementação que priorizam soluções rápidas e fáceis em custo de soluções mais robustas e sustentáveis. Este "débito" pode levar a problemas futuros, como maior dificuldade de manutenção, bugs e necessidade de refatoração. É importante gerir a Technical Debt para garantir a qualidade e a longevidade do software.

#### 2

a) A modularidade consiste em dividr um sistema de software em componentes ou módulos independentes, cada um responsável por uma funcionalidade específica. Isto facilita a manutenção, reutilização de código e colaboração entre equipas, permitindo ainda a escalabilidade e testagem do sistema.

b) A modularidade permite que seja desenvolvido um sistema de software escalável, pois cada módulo pode ser desenvolvido e testado indepentemente. Também facilita a adição de novas funcionalidades, uma vez que novos módulos podem ser integrados ao sistema existente sem afetar significativamente os módulos já existentes.

c) Uma DSM permite visualizar as dependências entre módulos de um sistema, ajudando a identificar acoplamentos desnecessários e áreas onde a modularidade pode ser melhorada.

d) Information Hiding é um principio que envolve expor apenas os detalhes estáveis de um módulo e colocar, num módulo separado, os detalhes que são mais propensos a mudanças. Isto reduz o impacto das mudanças em outras partes do sistema, melhorando a manutenibilidade, pois outros módulos só irão depender dos detalhes estáveis.

e) Para a análise de dependências implicitas e evolução arquitetural de um sistema, já que deteta acomplamento lógico, identifica problemas de modularização, suporta a refatoração e ajuda na compreensão da estrutura do sistema.

### G2

1) Para este tipo de sistema, uma arquitetura Pipe-And-Filter seria adequada. Esta arquitetura permite que os dados sejam processados em etapas sequenciais, onde cada etapa (filtro) transforma os dados e os passa para a próxima etapa (pipe). Isso é ideal para sistemas de processamento de dados, como sistemas de análise de logs ou processamento de imagens. Contudo, devido a sua natureza, existem problemas de latência e throughput.

3) O uso de múltiplas vistas arquiteturais é importante porque diferentes stakeholders têm diferentes preocupações e necessidades. Por exemplo, desenvolvedores podem estar interessados na vista de implementação, eqquanto que stakeholders de negócios podem estar mais focados na vista de processos. Utilizando múltiplas vistas, é possível comunicar efetivamente a arquitetura do sistema para todos os envolvidos, garantindo que todas as preocupações sejam abordadas.

### G3

#### 1

a) Um cenario de qualidade e uma descricao detalhada de uma situacao especifica que ilustra como um sistema deve se comportar em relacao a um atributo de qualidade, como desempenho, segurança ou usabilidade. Ajuda a eliminar ambiguidade, comunicacao entre stakeholders, apoiam decisoes de design, permitem a validacao e teste, priorizar requisitos e uma base para formas de avaliacao.

b) Data Accuracy:

- Fonte de Estimulo: fluxo de dados externos.
- Estimulo: chegada de dados continuos.
- Ambiente: operacao normal, carga media-alta.
- Artefato: modulo de processamento de dados.
- Resposta: aproximacao, atualiza dados em tempo real usando dados recebidos.
- Medido resposta: erros de dados baixa.

c) Condicionalmente, sim. Processados em memoria com o calculo feito de forma incremental ou por janelas temporais. PEdidos de leitura nao bloqueiam a pipeline, e a aproximacao mantem o erro dentro dos limites.

d) Faz sentido, contudo existem outras solucoes, como a modificacao dos algoritmos usados, desacoplamento de componentes, caching, particionamento de dados, e paralelismo. O aumento de recursos computacionais pode ser uma solucao temporaria, mas nao aborda as causas subjacentes dos problemas de desempenho.

### G4

#### 1

a) Uma DSL e uma linguagem de programacao especifica a um dominio particular, projetada para resolver problemas ou expressar conceitos dentro desse dominio de forma mais eficiente do que linguagens de uso geral. Exemplos incluem SQL para consultas de bases de dados, HTML para estrutura de paginas web, e VHDL para design de hardware.

b) Uma DSL externa e uma linguagem independente que e projetada para ser usada fora do contexto de uma linguagem de programacao existente (SQL, CSS). Ja uma DSL interna e incorporada dentro de uma linguagem de programacao hospedeira, aproveitando sua sintaxe e recursos (RubyOnRails, jQuery).
