### G1

#### 1. A maleabilidade do software é um dos vários fatores que colocam desafios e dificuldades únicas ao design de software

##### a. Explique porquê
>
>A maleabilidade representa a capacidade de um sistema de software adaptar-se a mudanças nos requisitos ou no ambiente operacional sem a necessidade de grandes modificações na arquitetura ou no código. Isso é importante porque permite que o software evolua ao longo do tempo, atendendo às novas necessidades dos utilizadores e às mudanças tecnológicas, sem incorrer em altos custos de manutenção ou riscos de falhas. Contudo, incorre num aumento da complexidade, o que pode dificultar o design inicial, e a maior maleabilidade é propicia a modificações frequentes, o que pode levar a problemas de scope creep e volatilidade nos requisitos.

##### b. Em que consiste o Technical Debt? De que forma está este fenómeno relacionado com a maleabilidade do software?
>
>Technical Debt é um conceito que se refere ao custo acumulado de decisões de design ou implementação que priorizam soluções rápidas e fáceis em custo de soluções mais robustas e sustentáveis. Este "débito" pode levar a problemas futuros, como maior dificuldade de manutenção, bugs e necessidade de refatoração. Está relacionado com a maleabilidade do software porque um sistema com alta technical debt tende a ser menos maleável, dificultando a adaptação a mudanças e aumentando o custo de evolução do software ao longo do tempo.

#### 2. A modularidade é um conceito de qualidade aplicável ao design de um sistema de software

##### a. Explique em que consiste
>
>A modularidade consiste em dividir um sistema de software em componentes ou módulos independentes, cada um responsável por uma funcionalidade específica. Isto facilita a manutenção, reutilização de código e colaboração entre equipas, permitindo ainda a escalabilidade e testagem do sistema.

##### b. Indique, justificando, dois conceitos de qualidade dos sistemas de software diretamente relacionados com a modularidade dos seus designs
>
>A modularidade permite que seja desenvolvido um sistema de software escalável, pois cada módulo pode ser desenvolvido e testado indepentemente. Também facilita a adição de novas funcionalidades, uma vez que novos módulos podem ser integrados ao sistema existente sem afetar significativamente os módulos já existentes. A modularidade também contribui para a manutenibilidade, pois problemas podem ser isolados e corrigidos em módulos específicos sem impactar o sistema como um todo, como também para a maleabilidade, permitindo que o sistema se adapte a mudanças de requisitos com menor esforço.

##### c. Indique de que forma matrizes como a mostrada ao lado (uma Design Structure Matrix) podem ser usadas para analisar a modularidade de um design
>
>Uma DSM permite visualizar as dependências entre módulos de um sistema, ajudando a identificar acoplamentos desnecessários e áreas onde a modularidade pode ser melhorada. No caso de existir um alto acoplamento de dependências entre certos módulos, isso pode indicar que esses módulos estão fortemente acoplados, o que pode dificultar a manutenção e evolução do sistema.

##### d. O Information Hiding é uma abordagem, introduzida por David Parnas, para conceber as estruturas modulares do design de um sistema de software. De acordo com esta abordagem, como devem ser tomadas as decisões de modularização?
>
>Information Hiding é um principio que envolve expor apenas os detalhes estáveis de um módulo e colocar, num módulo separado, os detalhes que são mais propensos a mudanças. Isto reduz o impacto das mudanças em outras partes do sistema, melhorando a manutenibilidade, pois outros módulos só irão depender dos detalhes estáveis.

##### e. Em que medida poderá ser útil uma ferramenta que, recorrendo à história das versões de um sistema de software, é capaz de produzir matrizes DSM que também exibem o número de vezes que dois módulos mudaram juntos (i.e., no mesmo commit)?
>
>Para a análise de dependências implicitas e evolução arquitetural de um sistema, já que deteta acomplamento lógico, identifica problemas de modularização, suporta a refatoração e ajuda na compreensão da estrutura do sistema.

#### Pretende-se desenvolver um sistema que permita monitorizar as tendências no uso de palavras numa língua. O nome já foi escolhido — WordTrending e, grosso modo, os requisitos do sistema também já foram elicitados e são os descritos abaixo

#### O WordTrending deve ser capaz de calcular a popularidade das palavras de uma língua (ou seja, a frequência com que são usadas), a partir de um fluxo contínuo de dados — objetos JSON contendo um texto de tamanho limitado e uma timestamp. Os dados recebidos, que podem corresponder por exemplo a tweets ou sms, são recolhidos e agregados por um sistema externo. As timestamps é suposto formarem uma sequência monótona crescente

### O WordTrending deve produzir um fluxo contínuo de objetos JSON contendo as 10 palavras mais populares, bem como as 10 palavras mais populares da última semana e do último mês. Estes dados podem ser usados, por exemplo, para manter atualizado um quadro de popularidade (ver figura ao lado). O sistema deve analisar o texto dos dados de input de forma a determinar se está escrito na língua escolhida e ignorar os que não o estejam. Para efeitos de apuramento de popularidade, apenas strings de alfanuméricos que não sejam palavras consideradas comuns (como por exemplo artigos e proposições) são consideradas. Apesar de os dados produzidos pelo sistema deverem levar em conta os dados recebidos até ao momento, há alguma tolerância para pequenos desvios em relação ao valor exato (data inaccuracy) sendo privilegiado o tempo de resposta, i.e., a velocidade com que os dados de saída refletem os dados de entrada. O sistema irá correr num servidor dual-core mas poderá ser movido para um outro servidor de 8-cores se for necessário. O sistema deve ser facilmente modificável para acomodar ajustes e pequenas alterações na forma como a popularidade das palavras é calculada e no conteúdo e representação dos dados que produz

#### 1. Indique um estilo arquitetónico que considere apropriado para usar neste sistema. Descreva, em abstrato, esse estilo (nomeadamente o tipo de componentes e conectores, e o modelo de computação) e as principais vantagens e desvantagens que lhe estão associadas
>
>Para este tipo de sistema, uma arquitetura Pipe-And-Filter seria adequada. Esta arquitetura permite que os dados sejam processados em etapas sequenciais, onde cada etapa (filtro) transforma os dados e os passa para a próxima etapa (pipe). Isso é ideal para sistemas de processamento de dados, como sistemas de análise de logs ou processamento de imagens. Contudo, devido a sua natureza, existem problemas de latência e throughput.

#### 3. Explique porque se usam várias vistas para apresentar a arquitetura de um sistema e exemplifique com o WordTrending, indicando que outras vistas seria útil produzir neste caso e porquê
>
> O uso de múltiplas vistas arquiteturais é importante porque diferentes stakeholders têm diferentes preocupações e necessidades. Por exemplo, desenvolvedores podem estar interessados na vista de implementação, eqquanto que stakeholders de negócios podem estar mais focados na vista de processos. Utilizando múltiplas vistas, é possível comunicar efetivamente a arquitetura do sistema para todos os envolvidos, garantindo que todas as preocupações sejam abordadas.

### G4

#### 1

#### a. A que se chama uma linguagem específica de domínio (DSL)? Dê dois exemplos
>
>Uma DSL e uma linguagem de programacao especifica a um dominio particular, projetada para resolver problemas ou expressar conceitos dentro desse dominio de forma mais eficiente do que linguagens de uso geral. Exemplos incluem SQL para consultas de bases de dados, HTML para estrutura de paginas web, e VHDL para design de hardware.

#### b. O que distingue as DSL internas e as externas? Dê um exemplo de cada
>
>Uma DSL externa e uma linguagem independente que e projetada para ser usada fora do contexto de uma linguagem de programacao existente (SQL, CSS). Ja uma DSL interna e incorporada dentro de uma linguagem de programacao hospedeira, aproveitando sua sintaxe e recursos (RubyOnRails, jQuery).
