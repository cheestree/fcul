### 1. Consider the following Java programs

```java
public class Producer {
    private static final int MAX = 10;
    public static void main(String[] args) throws InterruptedException {
        for(int i = 0; i<MAX; i++){
            if (i%2 == 0)
                System.out.println("AAA");
            else
                System.out.println("BBB");
            Thread.sleep(1000);
        }
        System.out.println("FIM");
    }
}
public class Consumer {
    public static void main(String[] args) {
        String token="";
        Scanner sc = new Scanner(System.in);
        do{
            token = sc.nextLine();
            System.out.print(token.charAt(0)+" ");
        } while(!(token.equals("FIM")));
        sc.close();
    }
}
```

#### a. What is the result of executing the following command in Unix: $ java Producer | java Consumer and what is the architectural style of the producer-consumer system defined in this way? Produce a module view and a component and connector view of this system
>
> The result of executing the command is that the output generated by the Producer program is piped as input to the Consumer program. The Producer generates a sequence of "AAA" and "BBB" strings, which are then read by the Consumer, which prints the first character of each string followed by a space. The architectural style of this system is a Pipe-and-Filter architecture, where the Producer acts as a filter that produces data and the Consumer acts as another filter that consumes data, with a pipe connecting them to facilitate data flow.

#### b. Without making any changes to the given programs, build a producer-consumer system with the batch sequential style. Produce a module view and a component and connector view of this system
>
> To build a producer-consumer system with the batch sequential style using the given programs, we can modify the execution flow to first run the Producer program to completion, capturing its output into a temporary file. Once the Producer has finished generating all the data, we can then run the Consumer program, providing it with the temporary file as input. This way, the Producer and Consumer operate in a sequential manner, with the Consumer only starting after the Producer has completed its task.

#### c. Regarding the properties of the two previous systems, what are the most important differences?
>
> The most important differences between the two systems are related to their execution flow and data handling. In the Pipe-and-Filter architecture, the Producer and Consumer operate concurrently, allowing for real-time data processing and a continuous flow of data. This can lead to lower latency and more efficient resource utilization. In contrast, the batch sequential style requires the Producer to complete its task before the Consumer can begin processing, which can introduce delays and increase overall processing time. Additionally, the batch sequential approach may require additional storage for intermediate data, while the Pipe-and-Filter approach processes data on-the-fly without needing to store it temporarily.

#### a. What are the two main styles used in the architecture of ETL systems?
>
>
> The two main styles used in the architecture of ETL (Extract, Transform, Load) systems are the traditional ETL process and the ELT (Extract, Load, Transform) process. In the traditional ETL process, data is extracted from source systems, transformed into a suitable format, and then loaded into the target data warehouse. In contrast, the ELT process involves extracting data from source systems, loading it directly into the target data warehouse, and then performing transformations within the data warehouse itself.

#### b. What do these two styles have in common?
>
> Both styles share the common goal of moving data from source systems to a target data warehouse for analysis and reporting. They both involve the key steps of extraction, transformation, and loading, although the order and location of these steps differ between the two styles. Additionally, both approaches aim to ensure data quality, consistency, and integrity throughout the ETL process.

### 4. Consider the following Java program that implements a very simple producer- consumer system

```java
public class SimpleSystem{
    public static void main(String[] args) throws InterruptedException {
        try {
            PipedReader reader = new PipedReader();
            Consumer consumerThread = new Consumer (reader);
            PipedWriter writer = new PipedWriter();
            Producer producerThread = new Producer (writer);
            writer.connect(reader);
            producerThread.start();
            consumerThread.start();
            consumerThread.join();
            for (char c: consumerThread.getChars()) System.out.print(" " + c);
        }catch (IOException e) {
            System.out.println ("Input/output error: " + e);
        }
    }
}
public class Producer extends Thread {
    Writer stream;
    Producer (Writer p) {
        stream = p;
    }
    public void run() {
        System.out.println ("Producer started... ");
        try {
            for (int i = 0; i < 9; i++) {
                char c = (char)(Math.random()*26 + 'a');
                stream.write (c); // producing
                System.out.println (i + ". produced: " + c);
                try {
                    sleep ((int)(Math.random()*1000));
                } catch (InterruptedException e) {
                }
            }
            stream.close(); // reader can continue reading
        }
        catch (IOException e) {
            System.out.println ("Input/output error: " + e);
        }
        System.out.println ("Producer finished...");
    }
    }
public class Consumer extends Thread {
    Reader stream;
    char[] chars;
    Consumer (Reader p) {
        stream = p;
        chars = new char[100];
    }
    public void run() {
        System.out.println ("Consumer started... ");
        try {
            int i = 0, r;
            while ((r = stream.read()) != -1) {
                System.out.println (i++ + ". consumed: " + (char)r);
                if (i<100) chars[i]=(char)r;
                try {
                    sleep ((int)(Math.random()*1000));
                } catch (InterruptedException e) {}
            }
            stream.close(); // reader closes finally
        }
        catch (IOException e) {
            System.out.println ("Input/output error: " + e);
        }
        System.out.println ("Consumer finished...");
    }
    public char[] getChars(){
        return chars;
    }
}
```

#### a. From an architectural standpoint, what can you say about the system?
>
> It's a Pipe-and-Filter architecture, where the Producer and Consumer are the filters, and the PipedReader and PipedWriter act as the pipe connecting them. The Producer generates data (characters) and writes them to the pipe, while the Consumer reads the data from the pipe and processes it.

#### b. Using the same Consumer and Producer classes, implement a version of the system with the batch-sequential style
>
>
```java
public class BatchSequentialSystem{
    public static void main(String[] args) throws InterruptedException {
        try {
            PipedReader reader = new PipedReader();
            Consumer consumerThread = new Consumer (reader);
            PipedWriter writer = new PipedWriter();
            Producer producerThread = new Producer (writer);
            writer.connect(reader);
            producerThread.start();
            producerThread.join(); // wait for producer to finish
            consumerThread.start();
            consumerThread.join();
            for (char c: consumerThread.getChars()) System.out.print(" " + c);
        }catch (IOException e) {
            System.out.println ("Input/output error: " + e);
        }
    }
}
```

> In this version, the main difference is that we wait for the Producer thread to finish using `producerThread.join()` before starting the Consumer thread. This ensures that the Producer completes its task of generating all the data before the Consumer begins processing it.

### 5. Consider the following Unix commands

```bash
mkfifo fifo1
mkfifo fifo2
grep TP12 < fifo1 > fifo2 &
cat file.txt | sort | tee fifo1 | grep TP16 | cat – fifo2 | uniq -c
```

#### a. What is the result of executing these commands?
>
> The result of executing these commands is that a named pipe (FIFO) called fifo1 is created, followed by another named pipe called fifo2. The `grep TP12 < fifo1 > fifo2 &` command runs in the background, continuously reading from fifo1 and writing lines containing "TP12" to fifo2. The final command `cat file.txt | sort | tee fifo1 | grep TP16 | cat – fifo2 | uniq -c` reads the contents of file.txt, sorts the lines, and uses `tee` to write the sorted output to fifo1 while also passing it to `grep TP16`. The lines containing "TP16" are then combined with the output from fifo2 (which contains lines with "TP12") and passed to `uniq -c`, which counts the occurrences of each unique line.
