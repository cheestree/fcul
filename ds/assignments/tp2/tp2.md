### 1. Technical debt is a relevant concept in software engineering

#### a. Explain what it consists of and give an example

> Technical debt is the growing cost of additional rework caused by choosing an easy solution now instead of using a better approach that would take longer. It accumulates over time as quick fixes and suboptimal solutions are implemented, leading to increased complexity and maintenance challenges.

#### b. What are the implications of technical debt for software systems and their development over time?

> Technical debt can affect software projects by slowing down development, increasing the likelihood of bugs, and making it harder to implement new features. It can lead to higher costs and longer timelines as more effort is required to address issues caused by the debt.

#### c. Some developers are beginning to use AI-generated code in their projects. In what ways could the use of AI-generated code increase technical debt in software systems?

> By using AI-generated code in projects, developers might introduce technical debt if the generated code is not well understood, poorly integrated, or lacks proper documentation. This can lead to maintenance challenges and increased complexity in the long run.

### 2. Consider the roles that different forms of prototype could perform (if any) in the development of the following software systems

### In each case, which group of people would be involved with reviewing any prototypes that were produced and what forms might a prototype usefully take? Which form of development process would be more suitable for each case?

#### a. an interactive word processor

> An incremental process for an interactive word processor, as users can request a new feature or improvement, and developers can implement it in small, manageable increments. Each increment can be tested and refined based on user feedback.

#### b. a web-based e-commerce site dedicated to selling sheet music of all types

> A linear process for a web-based e-commerce site dedicated to selling sheet music, as the requirements are likely well-defined and the project can follow a sequential approach from design to deployment without frequent changes.

#### c. a bank ATM for dispensing cash and providing receipts

> A linear process for a bank ATM software system, as the requirements are typically stable and well-understood, allowing for a structured approach to development.

#### d. a mobile social media application for sharing short videos

> A reactive process for a social media platform, as user needs and market trends can change rapidly, requiring the development team to adapt quickly to new requirements and feedback.

#### e. an open-source data visualization library

> A reactive process for an open-source project, as contributions from the community can lead to frequent changes and new features that need to be addressed promptly.

### 3. For every behaviour or functionality desired in a software system, numerous solutions exist. Take, for instance, the system responsible for calculating the income tax each Portuguese taxpayer must pay every fiscal year (IRS)

#### a. How will these solutions differ?

> Solutions differ on the basis of their architecture, technology stack, scalability, performance, and user experience. Each solution may prioritize different aspects such as speed, security, or ease of use.

#### b. Which solution should we opt for?

> We should opt for a solution that balances performance, scalability, and user experience while aligning with our project goals and constraints. It is important to consider long-term maintainability and the ability to adapt to future requirements.

#### c. To what extent does design differ from a search process?

> Design differs from a search process in the sense that design is a creative and iterative process focused on creating a solution, while a search process is more about exploring existing options to find the best fit. Design involves generating new ideas and concepts, whereas search involves evaluating and selecting from pre-existing alternatives.
