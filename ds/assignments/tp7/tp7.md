### 1. It is intended to develop a system that, based on data received from different sensors installed in a building in real-time, detects and reacts to problematic situations. The name has already been chosen — AliveBuildings — and, in broad terms, the system requirements have also been elicited and are described below

### AliveBuildings should be able to analyse the data received from various types of sensors installed in a building, such as emergency buttons, smoke detectors, motion detectors, presence detectors, and access control terminals. The analysis to be performed should aim to detect problematic situations such as fire, intrusion, and energy wastage. The system should be able to produce appropriate reactions to the detected situations using various types of actuators installed in the building, such as fire control panels, alarm sirens, lighting and heating system controllers, and information terminals (for example, used by building security). The system should be able to quickly detect the most dangerous situations and produce timely reactions. The system should also be easily modifiable to accommodate adjustments and changes in the characterization of problematic situations as well as in the way it reacts

#### a. Indicate an architectural style that you consider appropriate to use in this system. Describe, in abstract, this style (including the type of components and connectors, topological constraints, and the computing model) and the main advantages and disadvantages associated with it
>
> An appropriate architectural style for the AliveBuildings system could be the Event-Driven Architecture (EDA). In this style, components are designed to respond to events generated by sensors in real-time. The main components would include event producers (sensors), event consumers (analysis modules), and event brokers (middleware that routes events). The connectors would be event channels that facilitate communication between producers and consumers. The topology would be loosely coupled, allowing for scalability and flexibility in adding or modifying components. The computing model would be asynchronous, enabling the system to handle multiple events concurrently.
> Advantages of EDA include its ability to handle real-time data efficiently, scalability to accommodate more sensors and actuators, and flexibility in modifying the system. Disadvantages may include increased complexity in managing event flows and potential challenges in ensuring data consistency across components.

### 2. A spam prevention system is intended to be developed to process messages accepted by an email server. The system should analyse the content of messages, assuming that envelope-based checks are performed by the server. The goal is to detect viruses, identify dangerous files in attachments (e.g., cross-platform executables), and spot hazardous links. Infected messages should be destroyed; attachments deemed dangerous should be removed, and dangerous links should be converted to text and marked accordingly. The sender of an undelivered message or a message with removed attachments receives a notification about the incident

#### a. Choose an architectural style that you consider appropriate for use in this system, justifying your choice
>
> An appropriate architectural style for the spam prevention system could be the Pipe-and-Filter architecture. In this style, the system is composed of a series of processing components (filters) that transform the email messages as they flow through the system via connectors (pipes). Each filter would be responsible for a specific task, such as virus scanning, attachment analysis, and link checking. The modular nature of the Pipe-and-Filter style allows for easy addition or modification of filters as new threats emerge. This architecture also supports parallel processing, which can enhance performance when handling large volumes of email messages.

#### d. Identify the main advantages and disadvantages of using this style in the anti-spam system
>
> Advantages of the Pipe-and-Filter architecture include modularity, which allows for easy maintenance and updates to individual filters without affecting the entire system. It also supports parallel processing, which can improve performance when dealing with high volumes of email. Additionally, the clear separation of concerns makes it easier to understand and manage the system.
> Disadvantages may include potential latency introduced by the multiple processing stages, which could affect the speed of email delivery. There may also be challenges in managing the flow of data between filters, especially if the system needs to handle complex dependencies or stateful processing. Furthermore, debugging can be more difficult due to the distributed nature of the processing components.

### 3. Consider the design solution presented below for the anti-spam system

- Getter Emails: Receives the emails to be analyzed from the rcv emails queue; publishes an email event for each email, adding a unique identifier to the email.
- Virus Detector: Runs virus detection programs on each received email in an email event and publishes a virus event with the analysis result, along with the identifier of the email to which the result refers.
- Links Processor: Analyzes the links in the body of each received email in an email event, publishes a links event with the email in which the considered dangerous links have been transformed into text, along with the identifier of the email to which the result refers.
- Attachments Processor: Analyzes the attachments in each received email event. Publishes an attachs event with the email ID, names of files considered dangerous, and the email without those files in the attachments.
- Decision Maker: Decides what to do with each email based on the information
received in the links, virus, and attachs events: if a virus is detected, sends an email to the sender notifying that the email was not delivered; otherwise, places an email in the dlv emails queue with the body and attachments obtained, respectively, from the links and attachs events. In the case of removed files from the attachments, sends a notification email to the sender.

#### a. What is the architectural style of this solution? Describe the types of components, types of connectors, and the computing model of this style
>
> This is an Event-Driven Architecture (EDA) style. The components are event producers (Getter Emails, Virus Detector, Links Processor, Attachments Processor) and event consumers (Decision Maker). The connectors are event channels that facilitate communication between components. The computing model is asynchronous, allowing components to process events independently and concurrently.

#### b. Identify the main similarities and differences between the indicated style and the BlackBoard style
>
> They are both decoupled architectures that facilitate communication between components. They have a central coordination point, and process data asynchronously. In terms of differences, they differ in how data is shared and processed. In EDA, components communicate through events, while in Blackboard, components read from and write to a shared data structure (the blackboard). EDA is more focused on event handling, while Blackboard emphasizes collaborative problem-solving among components.

Disadvantages may include increased complexity in managing event flows and potential challenges in ensuring data consistency across components.
